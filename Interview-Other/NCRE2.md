#### C/S和B/S
+ C/S：客户端-服务器
  + 优点：
    + 个性化：客户端独立设计；
    + 响应速度快：充分发挥客户端PC的处理能力；
    + 安全性能保障：客户端是独立开发的；
  + 缺点：
    + 开发成本高：需要开发客户端和服务器两套程序；
    + 客户端程序设计复杂
    + 软件移植困难
    + 维护、升级成本高：客户端是不需要重复安装，所以用户可以不更新与升级；
+ B/S：浏览器-服务器
  + 优点：
    + 维护成本低：不需要安装客户端，只需要更新后台代码即可实现所有客户端的更新；
    + 开发成本低；
    + Web交互性强：
  + 缺点：
    + 耗流量，每次都要加载内容；
    + 系统扩展能力差，安全性难以控制；
    + 在数据查询等响应速度上慢于C/S；
    + 数据提交以页面为单位，数据动态交互性不强，不利于在线事务处理应用。
+ 区别：
  + 硬件环境不同
  + 安全要求不同
  + 程序架构不同
  + 软件重用不同
  + 系统维护不同
  + 处理问题不同
  + 用户接口不同
  + 信息流不同
+ C/S与B/S混合技术结构
  + “内外有别”模型：企业内部用户通过局域网直接访问数据库服务器（速度快），采用C/S；企业外部用户通过Internet访问web服务器，web服务器再服务数据库服务器（安全性高），采用B/S。
  + “查改有别”模型：凡是需要执行维护和修改数据操作的，使用C/S架构；如果只是执行一般的查询和浏览操作，使用B/S架构。

### 八、设计模式

#### 23种设计模式
+ 创建型模式
  + 单例模式
    + 优点
      + 可以在运行时添加或修改
      + 提供简化的创建结构
      + 具有给应用软件加载新功能的能力
      + 产品类不需要非得有任何事先确定的等级结构
    + 缺点
      + 每个类必须配置一个克隆方法
    + 适用场景
      + 在运行时，指定需要实例化的类，例如动态载入
      + 避免构造与产品的类层次结构相似的工厂类层次结构
      + 当类的实例是仅有的一些不同状态的组合
  + 工厂模式
  + 抽象工厂模式
    + 优点
      + 分离了具体类
      + 更容易在产品系列转换
      + 提高产品间一致性
     + 缺点
       + 难以支持新的产品结构
     + 适用场景
       + 系统独立于产品的创建、组成以及表示
       + 系统配置成具有多个产品的系列
       + 当要强调一系列相关的产品对象的设计以便于进行联合使用时
       + 当提供一个产品类库，而只想显示它们的接口而不是实现时
     + 应用举例：界面主题更换
  + 建造者模式
    + 优点
      + 产品的内部表象可以独立的变化
      + 将构造代码与表示代码相分离，可以使客户端不必知道产品内部组成的细节
    + 缺点
      + 如果产品之间差异性大，不适合使用
      + 如果产品内部变化复制，不适合
    + 适用场景
      + 创建复杂对象的算法独立于组成对象的部分以及这些部分的集合方式
      + 构造过程必须允许以构建对象有不同=表示
    + 应用举例：游戏软件中，地图人、人物的设计；
  + 原型模式
+ 结构型模式
  + 适配器模式
  + 装饰器模式
    + 优点
      + 灵活性
      + 简化代码
      + 改进扩展性
      + 装饰类、被装饰类独立发展，不耦合
    + 缺点
      + 多层装饰复杂
    + 适用场景
      + 想在单个对象中动态添加职责
      + 无法通过静态子类化实现扩展时
  + 代理模式
  + 外观模式
  + 桥接模式
    + 优点
      + 可以将接口与实现相分离
      + 提高可扩展性
      + 对客户端隐藏了实现的细节
    + 缺点
      + 增加了系统的理解和设计难度
      + 要求正确识别系统中两个独立变化的维度，因此适用范围有一定局限性
    + 适用场景
      + 想避免在抽象及其实现之间存在绑定
      + 抽象及其实现可以使用子类扩展
      + 抽象的实现被改动应该对客户端没有影响
  + 组合模式（整体模式）
  + 享元模式
    + 优点
      + 减少对象数目
      + 减少内存和存储设备
    + 缺点
      + 复杂化
      + 享元对象状态外部化，读取运行时间变长
    + 适用场景
      + 应用程序使用大量对象
      + 对象数目巨大，导致很高的内存开销
      + 应用程序不依赖对象的身份
+ 行为型模式
  + 策略模式
  + 模板方法
  + 观察者模式（发布—订阅模式）（事件监听模式）
    + 优点
      + 抽象了主体和观察者之间的耦合关系
      + 支持广播
    + 适用场景
      + 对一个对象修改涉及其他对象的修改，二不知道有多少个对象需要修改
      + 对象应该在不用假设对象标识的前提下通知其他对象
  + 迭代子模式
  + 责任链模式
  + 命令模式
  + 备忘录模式
  + 状态模式
  + 访问者模式
  + 中介者模式
  + 解释器模式
    + 优点
      + 容易修改并扩展语法
      + 容易实现语法
    + 适用场景
      + 语言的语法比较简单
      + 效率并不是最重要的问题
